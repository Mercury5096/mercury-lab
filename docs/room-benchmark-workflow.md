# Illustrated Room Benchmark Workflow

## Purpose

Use this workflow to develop the remaining Mercury Lab departments one room at a time. It is
derived from the `Cookbook Production Floor` and `AI Workflow Lab` desktop benchmarks, which
proved that a room can carry its own working identity while remaining inside one shared factory
cutaway language.

## Fixed Rules

- Begin with the approved room purpose and content boundary; do not invent stronger claims.
- Keep factual room copy and dossier access in HTML and driven by `src/content.js`.
- Use a cool, pale architectural cutaway background for fixed room structure.
- Reserve warm gold treatment for optional exploration: selected, movable, reviewed, or active items.
- Treat supplied reference images as design direction only unless explicitly cleared as public assets.
- Work desktop first. Do not restructure all floors or finalize compact layouts from one room.

## Room Design Sequence

### 1. Define The Room Story

Write the one-sentence purpose, the safe public boundary, and the room-specific workflow. A room
does not need to match another room's number of stages: Cookbook uses four practical actions;
AI Workflow Lab uses five decision stages.

### 2. Map Spatial Zones

Identify the work zones that must be legible before interaction. For each zone, define:

- Its production function.
- Its fixed architectural anchor: wall, table, workstation, shelf, console, or stage.
- The empty or subdued location reserved for an interactive object or signal.

### 3. Create The Fixed Background

Generate or illustrate a clean desktop background containing architecture and non-active
furnishings only. Avoid baking in prominent gold props, explanatory text, or effects that would
compete with later exploration layers.

### 4. Integrate At Actual Page Scale

Place the fixed background inside the existing room component before making props. Check:

- Whether the room needs a full-width single-floor presentation.
- Whether labels, route overlays, or the dossier cover important anchors.
- Whether the background should be compressed to WebP after the composition is accepted.

### 5. Prototype Exploration Before Producing Props

Test target location and behavior with lightweight CSS/SVG layers first when an interaction is
informational or diagrammatic. Generate transparent illustrated props only when material
texture makes the object meaningfully more convincing.

Each target should answer three questions:

- What does the user discover by hovering?
- What single action occurs on click?
- How does the action reinforce this room's real working method?

### 6. Review Interaction Boundaries

Pointer-led props are acceptable for optional desktop exploration when the factual room
description remains available through the keyboard-accessible dossier. If an interactive
element later contains unique or necessary information, it must receive an accessible
equivalent before approval.

### 7. Validate Before Expansion

For each benchmark:

- Inspect the room at desktop size with the dossier closed and open.
- Test each interaction and confirm return to the idle state.
- Check a compact breakpoint only for layout damage; defer detailed responsive design.
- Run the production build.
- Confirm formal content and rights-control source files were not changed.

Only after multiple rooms pass this process should the full building structure or downstream
sections be broadly revised.

## Next Room Selection Criteria

Choose the next department to test a different visual problem from the previous benchmarks:

- Distinct spatial character.
- Safe imagery boundary.
- Clear room-specific interaction metaphor.
- Useful evidence for deciding the final one-department-per-floor desktop architecture.

After Cookbook's physical publishing workflow and AI Lab's abstract decision workflow,
`Story Archive & IP Office` is the strongest next benchmark: it can test shelves, manuscripts,
rights-review folders, and creator handoff without repeating either kitchen objects or data nodes.
